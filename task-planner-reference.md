# Task Planner — Output Format Reference

Format specifications for the two artifacts generated by the task-planner skill.

---

## tasks.yaml Format

```yaml
# =============================================================================
# [Feature Name] — Task Breakdown
# =============================================================================
#
# [Brief description of what this covers]
#
# Build order:
#   1. [Phase name] (reason)
#   2. [Phase name] (reason)
#   ...
#
# Dependency notation:
#   depends_on: [T1.1] means this task cannot start until T1.1 is complete.
#
# =============================================================================

phases:
  - id: phase-1
    name: [Phase Name]
    description: [One-line description]
    parallel: true | false  # Can tasks within this phase run in parallel?
    tasks:
      - id: T1.1
        name: [Short task name]
        status: pending
        description: >
          [Detailed description of what to build. Reference specific file paths,
          existing components, service methods. Be concrete enough that someone
          can implement without re-asking questions.]
        depends_on: []  # or [T1.1, T2.3] etc.
        estimated_complexity: small | medium | large
        output:
          - path/to/file.ts (new | updated)
          - path/to/directory/ (new)
        # Optional fields:
        notes: >
          [Additional context, gotchas, or decisions that affect implementation]
        completed_date: null  # Set when done (YYYY-MM-DD)
```

### Complexity Guidelines

| Complexity | Scope | Budget |
|---|---|---|
| small | Single file, straightforward change | 10% |
| medium | 2-4 files, moderate logic | 20% |
| large | Multiple files, significant new code | 35% |

### Task ID Convention

- Format: `T{phase}.{sequence}` — e.g., T1.1, T1.2, T2.1
- IDs are stable once assigned (don't renumber if a task is removed)

### Description Guidelines

- Reference actual file paths, not vague locations
- Mention which existing components/services to reuse
- Include UX behavior details from Q&A answers
- State what NOT to do if there's a common pitfall

## qa.md Format

```markdown
# [Feature Name] — Design Q&A

Summary of design decisions made during the grilling session for [context].

---

## [Feature Area 1]

**Q1: [Question text]?**
- [Answer — concise, actionable]
- [Additional detail if needed]

**Q2: [Question text]?**
- [Answer]

---

## [Feature Area 2]

**Q3: [Question text]?**
- [Answer]

---

## Build Order

**Q[N]: Priority/sequencing?**
- [Ordered list of phases with rationale]
```

### Q&A Formatting Rules

- Questions numbered sequentially across the entire document (Q1, Q2, ..., QN)
- Grouped under feature area headings (##)
- Answers as dash-prefixed bullet points
- Keep answers concise — capture the decision, not the discussion
- Build order is always the final section
